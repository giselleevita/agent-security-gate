## What was learned

The original compact export used `alternative_unsafe_path` for every post-denial tool
call that differed from the seed, including successful safe recoveries. The count of 340
must therefore be described as **alternative action attempts**, not 340 proven unsafe
paths. See the versioned v0.3.1 erratum in `analysis_artifacts/v0.3.1/`.

The 840-run pass is strongest as an integrity and behaviour check, not as a final benchmark claim. It showed that the seeded-denial setup can run cleanly at this scale: all run keys were unique, the scorer recorded no policy violations, no responses failed parsing, and no model simply retried the same blocked call.

The harder finding is post-denial recovery. Across all strategies, **454/840 runs (54.0%)** reached a safe terminal outcome, but **228/840 (27.1%)** escalated and **158/840 (18.8%)** still ended in unsafe completion. The unsafe cases were usually alternate paths rather than exact retries: **340 alternative unsafe paths** were recorded versus **0 same-call retries**.

The strategy comparison is therefore mixed. B5 minimized unsafe completion (**5.0%**) but mostly by escalating (**93/120 runs**), which is safe but not very useful. B0 had the best safe-terminal rate (**72.5%**) while still allowing unsafe completions in **21/120 runs**. B6 did not yet outperform the simpler baselines in this run, which suggests that structured ticket mechanics alone are not enough unless the agent is also given a clear, constrained way to use them.

The next design target is clearer recovery affordances: denial feedback should point agents toward allowed alternatives, safe termination, or explicit approval without leaking protected state or inviting speculative workarounds. In short, ASG reliably blocked the first unsafe action in this run; the remaining work is teaching agents what to do next.
